# ADR 004: E2E テストに Playwright を採用

## 背景

これまでテストは Vitest による単体テスト（usecase / infrastructure / app など）に限られていた。
単体テストはモックで外部依存を切り離すため、次のような「画面をまたぐ実際の挙動」は検証できない。

- ランディングページの導線（CTA からの遷移）
- 認証画面の表示・遷移
- 未ログイン時のルート保護（`proxy.ts` のミドルウェアによる `/login` リダイレクト）

ブラウザを実際に動かし、ユーザー操作に近い形で回帰を検出する E2E テストを導入したい。

## 決定内容

E2E テストフレームワークとして **Playwright（`@playwright/test`）** を採用する。

- テストは `e2e/` 配下に配置（Vitest は `src/**/*.test.ts` のみ対象のため棲み分けできる）
- `playwright.config.ts` の `webServer` で `npm run dev` を起動し、ローカルの dev サーバーに対して実行する
- まずは外部バックエンド（Supabase / S3）に依存しない範囲（公開ページ・認証画面・ルート保護）から開始する

## 結果・利点

- **Next.js との親和性**: Next.js 公式ドキュメントでも E2E の選択肢として案内されており、App Router でも問題なく動作する
- **マルチブラウザ対応**: Chromium / Firefox / WebKit を 1 つの API で扱える（本プロジェクトは当面 Chromium のみ運用）
- **自動待機（auto-waiting）**: 要素の表示・操作可能状態を自動で待つため、フレーキーになりにくい
- **デバッグ体験**: `--ui` モード、トレースビューア、失敗時スクリーンショット/トレースが標準装備
- **CI 連携が容易**: 公式 Docker イメージや GitHub Actions 向けの導線が整っている

## 考慮点・トレードオフ

- ブラウザバイナリのダウンロードが必要（`npx playwright install`）。CI ではキャッシュ or 公式イメージの利用を検討する
- dev サーバーの起動が前提のため、単体テストより実行時間が長い
- 認証後フロー（レシピ登録など）の検証には、テスト用 Supabase プロジェクトやテストユーザー、データ投入の戦略が別途必要（本 ADR のスコープ外。今後拡張）
- `proxy.ts` が `NEXT_PUBLIC_SUPABASE_URL` 等を参照するため、E2E 実行時も `.env.local`（ダミー可）が必要

## 他ツールとの比較

### Cypress

- 実績が豊富で学習リソースも多い
- ただしブラウザ実行モデルの制約（同一オリジン制限など）があり、複数タブ・複数オリジンの扱いが Playwright より弱い
- WebKit 対応や並列実行は有料機能・追加設定に依存する部分がある
- Playwright の方が並列実行・マルチブラウザ・自動待機の総合力で優位と判断し見送り

### Selenium / WebdriverIO

- 歴史が長く対応ブラウザも広いが、セットアップ（ドライバ管理など）と記述量が多い
- モダンな DX（トレース・UI モード・自動待機）の面で Playwright に劣る
- 本プロジェクト規模ではオーバーヘッドが大きいため見送り

### Vitest + Testing Library（jsdom）での代替

- すでに採用済みだが、jsdom は本物のブラウザではなくレンダリング/ナビゲーションの差異がある
- ミドルウェアによるリダイレクトなど「サーバー＋ブラウザ」の結合挙動は検証できない
- 単体テストの補完として E2E が必要なため、置き換えではなく併用とする
