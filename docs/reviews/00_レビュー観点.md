# コードレビュー観点

このドキュメントでは、本プロジェクトのコードレビューで確認すべき観点をまとめています。

---

## 1. アーキテクチャ（クリーンアーキテクチャ）

### 1.1 依存方向

| チェック項目 | 詳細 |
|-------------|------|
| 依存は内側向きか | `app → usecase → domain` の方向を守っているか |
| usecase が infrastructure を import していないか | 実装ではなくインターフェースに依存しているか |
| domain が他層を import していないか | models, repositories は純粋な型定義のみか |

### 1.2 層の責務

| 層 | 確認事項 |
|---|---------|
| domain/models | 型定義のみ。ビジネスロジックや外部依存がないか |
| domain/repositories | インターフェースのみ。実装が含まれていないか |
| infrastructure | DB アクセスのみ。ビジネスロジックが入り込んでいないか |
| usecase | オーケストレーションと検証のみ。UI の都合が入っていないか |
| app | deps の組み立てと usecase 呼び出しのみ。ビジネスロジックがないか |
| presentation | UI のみ。usecase や infrastructure を直接呼んでいないか |

## 2. 命名・可読性

### 2.1 命名規則

| 対象 | ルール | 例 |
|-----|-------|---|
| ファイル名 | ケバブケース | `create-recipe-usecase.ts` |
| 型・クラス | パスカルケース | `RecipeRepository`, `SignupUseCase` |
| 関数・変数 | キャメルケース | `createRecipeUsecase`, `recipeId` |
| 定数 | アッパースネークケース | `ERROR_MESSAGES` |

### 2.2 可読性

| チェック項目 | 詳細 |
|-------------|------|
| 関数は単一責任か | 1 つの関数が複数のことをしていないか |
| 関数の長さは適切か | 50 行を超えていたら分割を検討 |
| ネストが深すぎないか | 3 階層以上のネストは早期リターンで解消 |
| マジックナンバーがないか | 数値は定数に切り出す |

---

## 3. 型安全性（TypeScript）

| チェック項目 | 詳細 |
|-------------|------|
| `any` を使っていないか | 型が不明な場合は `unknown` を使う |
| 型アサーション（`as`）を多用していないか | 本当に必要か確認 |
| `null` / `undefined` を適切に処理しているか | オプショナルチェーン、Nullish coalescing を活用 |
| 戻り値の型を明示しているか | 関数の戻り値型は明示推奨 |

---

## 4. エラーハンドリング

| チェック項目 | 詳細 |
|-------------|------|
| try-catch が適切か | 握りつぶし（空の catch）がないか |
| エラーメッセージは適切か | ユーザーに見せるメッセージと開発者用ログを区別 |
| Result 型を活用しているか | `{ success: true, data } \| { success: false, error }` 形式 |
| 予期しないエラーを考慮しているか | 想定外のエラーも適切に処理 |

---

## 5. セキュリティ

| チェック項目 | 詳細 |
|-------------|------|
| 入力値の検証をしているか | フロントとサーバー両方で検証 |
| 認証チェックがあるか | 認証が必要な操作で `createAuthedClient` を使用 |
| 認可チェックがあるか | 自分のリソースのみ操作できるか確認 |
| SQL インジェクション対策 | Supabase のパラメータ化クエリを使用 |
| XSS 対策 | ユーザー入力をそのまま表示していないか |
| 環境変数の漏洩 | クライアントに公開していい値のみ `NEXT_PUBLIC_` を使用 |

---

## 6. パフォーマンス

### 6.1 フロントエンド

| チェック項目 | 詳細 |
|-------------|------|
| 不要な再レンダリングがないか | `useMemo`, `useCallback` の適切な使用 |
| コンポーネントの分割は適切か | 状態変更の影響範囲を最小化 |
| 画像の最適化 | `next/image` を使用しているか |

### 6.2 バックエンド

| チェック項目 | 詳細 |
|-------------|------|
| N+1 問題がないか | ループ内で DB アクセスしていないか |
| 不要なデータ取得がないか | `select()` で必要なカラムのみ取得 |
| 並列実行できる処理か | 独立した処理は `Promise.all` で並列化 |

---

## 7. テスタビリティ

| チェック項目 | 詳細 |
|-------------|------|
| 依存が注入可能か | usecase は deps パターンで DI できるか |
| 副作用が分離されているか | 純粋な関数とDB アクセスが分かれているか |
| モック可能か | 外部依存をモックしてテストできる構造か |

---

## 8. Next.js / React 固有

| チェック項目 | 詳細 |
|-------------|------|
| Server/Client Component の使い分け | `"use client"` は必要な箇所のみ |
| Server Actions の使い方 | `"use server"` が適切に付与されているか |
| フォーム送信 | `useActionState` または `startTransition` を使用 |
| リダイレクト | `redirect()` は try-catch 外で呼ぶか、エラーを再スロー |

---

## レビュー時の心構え

### やるべきこと

- 具体的な改善案を提示する
- なぜ問題なのかを説明する
- 良い点も積極的に伝える
- 学びになるポイントを共有する

### やってはいけないこと

- 人格を否定するコメント
- 好みだけを押し付ける（明確な理由がない指摘）
- 重箱の隅をつつくような指摘ばかり
