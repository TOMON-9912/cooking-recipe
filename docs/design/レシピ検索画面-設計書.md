# レシピ検索画面 設計書

## 0. この設計書の使い方と学習ゴール  

### 使い方

1. **セクション 1〜6** で「何を作るか」と全体像を把握する。
2. **セクション 10（実装ガイド）** に沿って、ステップ 1 から順に手を動かす。各ステップの「完了したら」を満たしてから次へ進む。
3. 行き詰まったらチャットでペアプロモードに質問する（答えのコピペではなく、**見る場所・考え方のヒント**をもらう）。

### 学習ゴール（この機能を通じて身につけたいこと）

- **依存の向き**: domain にインフラを引き込まない、usecase は deps 経由で処理を受け取る（`.cursor/rules/clean-architecture.mdc`）。
- **Next.js App Router**: サーバーコンポーネントの `page.tsx` でデータ取得し、URL（`searchParams`）と画面状態を対応させる。
- **既存パターンの踏襲**: トップのレシピ一覧と同じ「サマリー取得 → お気に入りマージ → サムネ URL 付与」の流れを、検索条件付きで再現する。

## 1. 概要

家族が登録した**公開済みレシピ**を、キーワードと（必要なら）カテゴリで絞り込み、一覧で見つけられる画面を追加する。トップのクイックアクセス「レシピ検索」（現状 `TopHero.tsx` の `QuickAccessSection`）から遷移できるようにし、**主目的は「条件を入れて結果を見る」1 つ**に絞る。

**スコープの前提（重要）**  
`recipe_summaries` ビューには **タイトル・説明・カテゴリ** はあるが、**材料名は含まれていない**（`docs/tables/view/recipe_summaries.md`）。文言上「食材から探す」とあるため、**第1段階は「タイトル・説明のキーワード + カテゴリ」**とし、**材料横断検索は第2段階**（ビュー拡張や別クエリ）として切り出すと実装が破綻しにくい。

## 3. 型定義

### 3.1 `recipe-search-query.ts`（例）

- `RecipeSearchQuery`（または同等の名前）: 1 回の検索の入力
  - `keyword: string` — 前後空白トリム後に使う。空なら「キーワード条件なし」
  - `categorySlug?: string` — 指定時のみカテゴリで絞る（`categories` の `slug` と一致）。未指定ならカテゴリ条件なし

※ URL の `searchParams` と 1:1 にできる形にしておくと、`page.tsx` が単純になる。

## 4. 「インターフェース」定義（依存の形）

既存の `get-recipe-summaries-usecase.ts` と同様、**domain に read 用 Repository ファイルが無いパターン**に合わせ、**usecase の `deps` が契約**とする。

### 4.1 `searchRecipeSummariesUsecase` の deps（例）

- `searchRecipeSummaries`: 引数に検索条件型、戻り値に `RecipeSummary` の配列を返す非同期関数 — インフラ実装が差し替え可能
- `getFavoriteRecipeIds`: 引数なしでお気に入りレシピ ID の配列を返す非同期関数 — 既存一覧と同じ

戻り値: `RecipeSummaryWithFavorite` の配列（`getRecipeSummariesUsecase` と揃える）

### 4.2 インフラ（Supabase）側の責務

- 常に `is_draft = false` のみ（既存 `getRecipeSummaries` と同じ公開範囲）
- キーワード: `title` / `description` に対する部分一致（PostgreSQL の `ilike` 相当）。**SQL インジェクションに注意**し、クライアントから来た文字列をそのまま文字列連結で SQL に埋め込まない（Supabase の API がどうパラメータ化するかを確認する）
- **キーワードのエスケープ（次の実装として推奨）**: `ilike` では `%` と `_` がパターン用のメタ文字になる。ユーザーが「50%」のように入力したときに意図しない広い一致にならないよう、**インフラ層**（`recipe-read-repository-impl.ts` 内のヘルパー、または `src/lib` の純関数をインフラからだけ import）で、パターン組み立て前にエスケープ処理を行う。**ドメイン・usecase では行わない**（DB 都合の詳細にしない）。具体的手順はセクション 10.3 補足 8 を参照。
- カテゴリ: `categories` が JSON 配列のため、**PostgREST のフィルタで書けるか**は実装時に要確認。難しければ「カテゴリ slug で recipe ID を取得するサブクエリ or 別テーブル結合」に切り替える、など設計の分岐をメモしておく

## 5. UI デザイン指針

- **Primary は 1 つ**: その画面では「検索する」（送信）を Primary（`bg-emerald-600` / `rounded-lg`）に。結果カード内のお気に入りはアイコンボタン扱いでよい（既存 `RecipeCard` に合わせる）。
- **ファーストビュー**: 検索窓 +（任意）カテゴリ、すぐ下に結果または空状態。
- **カラー**: `emerald-600` / `amber-500`（アクセント・バッジ）、テキストは `text-gray-900` / `700` / `500`。
- **空状態**: 説明文 + 次の行動（例: 条件をクリア、トップへ、`/recipe/new` は Secondary）。
- **角丸**: 入力 `rounded-md`、送信ボタン `rounded-lg`、結果カードは `RecipeCard` に合わせ `rounded-2xl`。
- **アイコン**: `lucide-react`（例: `Search`）。
- **画像**: サムネは既存どおり `next/image` + プレサイン URL（`top/page.tsx` と同様のパターン）。

**注意（ワークスペースルール）**  
`RecipeListPage` の空状態に絵文字が使われているが、**新規コードではコード内絵文字を増やさない**方針（`.cursor/rules/code-style.mdc`）。検索画面の空状態は `lucide-react` アイコン + テキストが無難。

## 6. ルーティングとデータ取得の流れ

1. **URL**: 例 `GET /recipe/search?q=カレー&category=japanese`（クエリ名は実装で統一）
2. **`page.tsx`**: `searchParams` を検索条件型に変換 → usecase 実行 → 各レシピの `thumbnailPath` にプレサイン URL 生成処理を適用（`top/page.tsx` の一覧取得部と同趣旨）
3. **フォーム**:
   - **推奨**: `method="get"` + `action="/recipe/search"` でサーバー再取得（ブックマーク・共有に強い）
   - 代替: クライアントでルーターを使い URL だけ更新 — 挙動は揃えれば可

## 7. 実装の流れ（推奨順・一覧）

推奨は **domain → infrastructure → usecase → app → presentation → 導線**（内側から外側へ形を作り、最後に画面をつなぐ）。

1. 検索条件の型（domain）
2. リポジトリ実装に検索用の関数を追加（infrastructure）
3. 検索 usecase（usecase）
4. ルート `page.tsx`（app）
5. 検索 UI コンポーネント（presentation）
6. トップからのリンク（TopHero）

## 10. 実装ガイド（初心者向け・ソース全文は書かない）

ここでは **どう進めるか** と **各ファイルに何を置くか** を文章で示す。コピペ用の完成コードは意図的に書かない（自分の手で書くことが学習になる）。

### 10.1 進め方の原則

- **1 ステップずつ**: 次の「完了したら」を満たしてから次へ。一度に複数レイヤーをいじると、不具合の原因が切り分けにくくなる。
- **既存を複製してから削る**: `getRecipeSummaries` と `getRecipeSummariesUsecase` の構造を頭に置き、「検索条件が増えた版」として読み替える。
- **型エラーを手がかりにする**: TypeScript のエラーは「依存の向き」や「引数の不足」を教えてくれる。消す前にメッセージを読む。

### 10.2 ステップ 1 — 検索条件の型（domain）

- **触るファイル**: `recipe-search-query.ts`（新規）
- **書く内容のイメージ**: URL やフォームから受け取りたい条件だけをフィールドに持つ型（または型エイリアス）。ここには **関数も DB も import しない**。
- **完了したら**: プロジェクトがビルドでき、他ファイルからこの型を import できる。

#### ステップ 1 補足 — ここまでのセルフチェック

- `keyword` を常に `string` にしている場合、**空文字**を「キーワード条件なし」とみなすか、インフラ側で **トリムしてから長さ 0 ならフィルタを付けない**か、どちらかに統一する（URL から来る値は空になりがち）。
- 設計上 `categorySlug` が無いときはカテゴリで絞らない。オプショナル `?` と、インフラでの分岐の対応を確認する。

### 10.3 ステップ 2 — DB から条件付きでサマリーを取る（infrastructure）

- **触るファイル**: `recipe-read-repository-impl.ts`（既存に関数を追加）
- **書く内容のイメージ**:
  - 新しい **名前付きの export 関数**（例: 検索用の名前）。引数はステップ 1 の型、戻りは既存 `getRecipeSummaries` と同じドメインの `RecipeSummary` 配列。
  - 中身の方針: 既存の `getRecipeSummaries` と同じテーブル（ビュー）を見る。そこに **公開済みフィルタ** をかけたうえで、キーワードがあるときだけタイトル・説明に部分一致、カテゴリ slug があるときだけカテゴリで絞る。
  - 行から `RecipeSummary` への変換は **既存関数と二重にならないよう**、共通部分を小さな内部ヘルパーに寄せてもよい（重複はバグの元）。
- **完了したら**: 仮で `page.tsx` から直接呼ばず、**コンソールログや一時的なテスト用の呼び出し**でもよいので、実際に Supabase から期待した件数が返ることを確認できる（ローカルで実行）。

#### ステップ 2 補足 — 詰まりやすい点（実装と照らしたヒント）

次は「完成コード」ではなく、**契約・名前・既存コードとの揃え**でほぼ解消できることが多い。

1. **関数の「入出力」を先に固定する**  
   - **第 1 引数**: ステップ 1 の型（例: `RecipeSearchQuery`）。レシピの `id` や別の意味の文字列にしない。  
   - **戻り値の型**: `Promise<RecipeSummary[]>`。検索条件オブジェクトを返すシグネチャになっていないか、エディタの戻り値型で確認する。

2. **参照するリソース名は既存の成功例と同じにする**  
   - 一覧取得で既に動いている `getRecipeSummaries` は **ビュー `recipe_summaries`** に対して `.select("*")` している。検索用関数も **まず同じ `.from(...)`** から始め、`is_draft` フィルタも同様にかけると、テーブル名の取り違え（例: `recipe` / `recipes` など）を防げる。  
   - ビュー定義と列名は `docs/tables/view/recipe_summaries.md` を参照。

3. **行 → `RecipeSummary` の変換は 1 か所に寄せる**  
   - `getRecipeSummaries` 内の `data.map((row) => ({ ... }))` と**同じ形**のオブジェクトを組み立てる必要がある。コピペで二重管理すると片方だけ直し忘れるので、**ファイル内の非 export の小さな関数**（例: 行を受け取り `RecipeSummary` を返す）にまとめると設計書の意図どおりになる。

4. **キーワード条件（タイトル・説明）**  
   - PostgreSQL では部分一致に `ilike` が使える。Supabase JS クライアントでは **`title` と `description` のどちらかに当てはまる**ようにするには、`.or(...)` などの書き方と、**パターン文字（例: 前後に `%`）**の渡し方を [Supabase のフィルタ API](https://supabase.com/docs/reference/javascript/using-filters) で確認する。  
   - **キーワードが空（条件なし）のときは、この OR 条件を付けない**（付けると意図と違う結果になりうる）。  
   - メタ文字の扱いは **補足 8**（エスケープを別タスクで足す）。

5. **カテゴリ slug 条件**  
   - ビューの `categories` は **JSON 配列**（`[{ id, name, slug }, ...]`）。PostgREST だけで「配列内の slug が一致」フィルタが難しい場合は、セクション 4.2 にあるとおり **別クエリで該当レシピ ID を求める**、または **開発中のみ**取得後にアプリ側で `slug` を走査して絞る、などに分岐してよい（本番前にデータ件数と合わせて見直す）。

6. **import の向き**  
   - infrastructure から検索条件型を import するときは、値として使わないなら `import type { RecipeSearchQuery }` にすると、バンドラ・リントと相性がよいことが多い。

7. **動かないときの切り分け順**  
   - 型エラー（戻り値が `{}` や不完全）→ まずシグネチャと `RecipeSummary` の全フィールドを満たしているか。  
   - 実行時エラー（relation does not exist 等）→ `.from` の名前が DB と一致しているか。  
   - 0 件だけ返る → `is_draft`・RLS・キーワード／カテゴリ条件の**どれかが厳しすぎないか**を、条件を外しながら試す。

8. **キーワードの `ilike` 用エスケープ（ステップ 2 の次に取り組む実装）**  
   - **目的**: トリム後のキーワードを `%...%` で包む前に、`%`・`_` をリテラルとして扱えるようにする（ワイルドカードとして解釈されないようにする）。必要に応じて、使用するエスケープ文字（例: PostgreSQL の `ESCAPE` と組み合わせる場合の `\`）も同様に処理する。  
   - **置き場所**: **インフラ層**が第一候補。ファイルが肥大化する場合は `src/lib/` に **純関数**として切り出し、**インフラからのみ** import する（domain / usecase からは import しない）。  
   - **やらないこと**: ドメインモデルに「エスケープ済み文字列」型を増やしてビジネスルールっぽく見せる、は過剰になりがち。入力は引き続き `RecipeSearchQuery.keyword` のまま受け、**クエリ組み立て直前**でだけエスケープする。  
   - **確認**: `%` や `_` を含むキーワードで、部分一致の挙動が「文字としての一致」になっているか手動で確認する。

### 10.4 ステップ 3 — お気に入りを付ける usecase（usecase）

- **触るファイル**: `search-recipe-summaries-usecase.ts`（新規）
- **書く内容のイメージ**:
  - ファイル先頭の `deps` 型: 「検索でサマリー配列を返す関数」と「お気に入り ID 一覧を返す関数」の 2 つ。
  - メインの exported 関数: 引数は `(検索条件, deps)` または `(deps, 検索条件)` など **プロジェクト内の他 usecase と同じ並び**に揃える。
  - 処理の流れは `getRecipeSummariesUsecase` と同型: 2 つの非同期処理を並列で待ち、ID の集合で `isFavorited` を付ける。
- **完了したら**: `getFavoriteRecipeIds` を既存実装に差し替えて import エラーがなく、戻り値の型が `RecipeSummaryWithFavorite` の配列になる。

### 10.5 ステップ 4 — ルートページ（app）

- **触るファイル**: `src/app/recipe/search/page.tsx`（新規）
- **書く内容のイメージ**:
  - Next.js のページコンポーネントが受け取る **`searchParams`** から文字列を取り出し、ステップ 1 の型に変換（空文字は「条件なし」にするなど、ルールを決める）。
  - `deps` オブジェクトを組み立て、ステップ 3 の usecase を呼ぶ（インフラの検索関数と、既存の `getFavoriteRecipeIds` を渡す）。
  - 返ってきた各レシピについて、トップページと同様に **サムネのプレサイン URL** を付ける処理を行う（非同期が増えるので、`Promise.all` などのパターンを既存から流用）。
  - エラー時はトップと同様にユーザー向けの簡単なエラー表示にするか、方針を決めて統一する。
- **完了したら**: ブラウザで `/recipe/search` を開き、エラーなく画面が表示される（結果ゼロでもよい）。

### 10.6 ステップ 5 — 検索 UI（presentation）

- **触るファイル**: `RecipeSearchPage.tsx` など（新規）
- **書く内容のイメージ**:
  - **props**: 検索結果の配列、現在のクエリ文字列・カテゴリ（フォームの初期表示用）、必要ならコールバック。
  - **フォーム**: セクション 6 の方針どおり GET 送信するか、クライアントで URL を更新するかを決め、入力名を `page.tsx` の `searchParams` のキーと一致させる。
  - **結果部**: `RecipeCard` を map で並べる。お気に入りトグルをトップと同じ体験にするなら、`RecipeListPage` と同様にローカル state で `isFavorited` を更新する仕組みを検討する。
  - **空状態**: 文言 + Secondary の「条件をクリア」「トップへ」など（`.cursor/rules/ui-design.mdc`）。
- **完了したら**: キーワードを入れて送信すると URL が変わり、結果が変わる（または 0 件表示が一貫する）。

### 10.7 ステップ 6 — トップからの導線

- **触るファイル**: `TopHero.tsx`
- **書く内容のイメージ**: 「レシピ検索」カードを `Link` で `/recipe/search` に繋ぐ。プレースホルダー用の `badge` の扱いを決める。
- **完了したら**: トップからクリックで検索ページに遷移する。

- [ ] 空状態と Primary が 1 つだけか（UI ルール）

### 10.9 つまずきやすい点（ヒントのみ）

- **カテゴリが JSON 配列**: DB 側のフィルタが難しければ、「一度広く取ってアプリ側で絞る」はデータ量次第では避けた方がよい。ドキュメントと PostgREST の能力を確認してから決める。
- **`searchParams` の型**: App Router のバージョンによって `searchParams` が Promise かどうかなど差がある。自分のプロジェクトの他 `page.tsx` の書き方に合わせる。
- **presentation と domain**: UI から domain の型を直接 import するか、`types` 経由にするかは既存コンポーネントに合わせる。

## 11. レイヤー早見表（何を import してよいか）

| 層             | ここでやること                    | import してよい例                                 | import してはいけない例                     |
| -------------- | --------------------------------- | ------------------------------------------------- | ------------------------------------------- |
| domain/models  | データの形だけ                    | なし、または他モデル型のみ                        | `supabase`、`react`                         |
| usecase        | 手順の組み立て、deps 型           | domain の型、（既存どおり）表示用の型があればそれ | infrastructure の実装ファイル               |
| infrastructure | Supabase 呼び出し、行→モデル変換  | domain の型、Supabase クライアント作成系          | usecase                                     |
| app            | deps の組み立て、usecase 呼び出し | usecase、infrastructure の **名前付き export**    | ビジネスルールの重複（なるべく usecase へ） |
| presentation   | 表示と入力                        | props 型、`RecipeCard`、`Link`、`lucide-react` 等 | 可能ならインフラ直叩きは避ける              |

（プロジェクトの厳密なルールは `clean-architecture.mdc` が最優先。）
