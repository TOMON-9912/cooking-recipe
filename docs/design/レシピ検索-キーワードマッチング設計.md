# レシピ検索：キーワードマッチングの設計（実装方針）

## 位置づけ

レシピ検索画面のうち、**キーワードでタイトル・説明を部分一致させる**部分の実装方針を記録する。画面全体の仕様は `レシピ検索画面-設計書.md` を参照する。

## 背景

- キーワード条件を **SQL の `ILIKE` とパターン文字（`%` で前後を囲む）** で表現すると、ユーザー入力の **`%` と `_` は `ILIKE` のメタ文字**として解釈される。
- 利用者が「50%」のように **記号を文字として**含む文字列を検索したい場合、**意図しない広い一致**になる、という問題がある。
- 設計書では **インフラ層でエスケープを検討**する旨が書かれていたが、**採用する実装手段**（アプリ内の文字列置換だけで足りるか）はスタック依存になる。

## 悩んだこと

### PostgREST / Supabase の `ilike` フィルタと `ESCAPE`

- PostgreSQL では、メタ文字をリテラルにしたい場合 **`ILIKE ... ESCAPE '\'`** のように **エスケープ句付き**で書くことが多い。
- アプリから渡すのは **パラメータ化された値**であり、**SQL インジェクション対策としてはクライアントの正しい利用が主**になる（文字列連結で生 SQL を組まないこと）。
- 一方、**`%` / `_` を「検索パターンのメタ文字にしない」**という話は、**LIKE 家族の意味論**の問題であり、**バックスラッシュを付けただけでは**、生成される SQL に **`ESCAPE` が付かない**限り、PostgreSQL 側の挙動が期待どおりにならないことがある。
- そのため、「アプリで `\` `%` `_` を置換して `ilike` に載せる」だけでは、**環境によっては不十分**になり得る、という判断をした。

### 「エスケープ」と「セキュリティ」

- **SQL インジェクション**の主な対策は、**パラメータ化**と **RLS** などであり、LIKE 用のエスケープそのものとは別次元の話が多い。
- 本件の「エスケープ」は、主に **検索結果の正しさ（ユーザーが記号を文字として探したい）** に効く。

## 検討した選択肢

### 選択肢 A：クライアント側でパターン用エスケープし、従来どおり `ilike` で検索

- **概要**: インフラ（または `src/lib` の純関数）で `\` `%` `_` をエスケープし、`%keyword%` 形式で PostgREST の `ilike` に渡す。
- **メリット**
  - DB マイグレーションを増やさずに済む場合がある。
  - 既存の「ビューに対して `select` + フィルタ」のパターンと揃えやすい。
- **デメリット**
  - 生成 SQL に **`ESCAPE` が付くか**は PostgREST / 利用 API に依存し、**挙動を確実にしたいときに検証コストがかかる**。
  - 予約文字や URL エンコーディングと **二重に気をつける**必要が出ることがある。

### 選択肢 B：`ILIKE` を使わず、部分文字列一致（`strpos` + `lower`）でマッチし、該当 ID を RPC で返す（採用）

- **概要**: DB に関数 `recipe_summaries_ids_matching_keyword(search_keyword text) returns uuid[]` を定義し、アプリは **`rpc` で ID 配列を取得**したうえで、`recipe_summaries` を **`id` の `in` で絞る**。カテゴリ条件がある場合は **キーワード結果 ID とカテゴリに紐づく ID の積集合**とする。
- **メリット**
  - ユーザー入力は **常に「普通の文字列」として比較**されるため、`%` `_` が **ワイルドカードにならない**（メタ文字のエスケープ問題を **設計で回避**できる）。
  - 「部分一致」の意味が **strpos ベースで明示的**になる。
- **デメリット**
  - **マイグレーション**が必要（関数定義と `GRANT`）。
  - キーワードでヒットする ID が **非常に多い**場合、**`.in('id', ids)` の URL 長や上限**がボトルネックになり得る（規模が大きくなったら別策を検討する余地あり）。
  - **インデックスを効かせにくい**（先頭一致でない限り、従来の `ILIKE '%...%'` と同様、負荷の話は残る）。

### 選択肢 C：全文検索や外部検索エンジン（将来）

- **概要**: `tsvector`、pg_trgm、Meilisearch / Elasticsearch など。
- **メリット**: 大規模・ランキング・日本語の扱いなど、**検索品質を上げやすい**。
- **デメリット**: インフラ・運用・学習コストが増える。現段階のスコープ（家族向け・第1段階）では **過剰**になりやすい。

## 最終決定

- **選択肢 B を採用**した。
- **理由の要約**
  - アプリだけで `ILIKE` 用エスケープを完結させるより、**「LIKE パターンを使わない」**方が、**記号の扱いを仕様として説明しやすい**。
  - Supabase 利用時に **RPC で DB 側に条件を閉じる**形にし、**クライアントは ID リストで絞り込む**責務に分けた。

## 運用上の注意

- 本番・ローカルの DB に **マイグレーション未適用**のままだと、`rpc` が存在せず **実行時エラー**になる。デプロイ手順にマイグレーション適用を含めること。
- 関数には `authenticated` への `EXECUTE` を付与している。匿名利用の要否が変わったら **権限を見直す**こと。

## 将来の伸ばし方（参考）

- 件数・負荷が増えたら **ページング**、**入力長の上限**、**レート制限**の見直し。
- 検索要件が高まったら **全文検索・トライグラム索引・外部検索**など、選択肢 C への段階的移行を検討する。
