# 実装モード

あなたは今から**実装モード**で動作します。

## このモードの目的

ユーザーの要求に対して、**実際にファイルを作成・編集**してコードを実装します。
ただし、**一度に大量の変更を出さず、レビュー可能な小さなスライス**で進めます。

## 動作ルール

1. **直接実装**: コードをファイルに書き込む
2. **段階的に進める**: 層ごと・1 スライスごとに実装し、各ステップ後に確認を取る
3. **クリーンアーキテクチャ遵守**: `.cursor/rules/clean-architecture.mdc` に従う
4. **命名規則遵守**: `.cursor/rules/file-naming.mdc` に従う
5. **コードスタイル遵守**: `.cursor/rules/code-style.mdc` に従う（絵文字・過度なコメント禁止）
6. **UIデザイン遵守**: `.cursor/rules/ui-design.mdc` に従う（統一感のあるデザイン）


## 実装規律（レビュー負荷を抑ぐための必須ルール）

### スライスの定義

**1 スライス = 次のいずれか 1 つだけ**を実装する。

| 層 | 1 スライスの例 |
|---|---|
| `domain/models` | 1 関心事の型定義 |
| `domain/repositories` | 1 リポジトリ interface の追加・変更 |
| `infrastructure` | リポジトリ実装の 1 メソッド（または密接な少数メソッド） |
| `usecase` | 1 ユースケース関数 |
| `app` | 1 Server Action |
| `presentation` | 1 画面、または 1 コンポーネント群 |

- **禁止**: 複数層をまたぐ実装、機能全体の一括実装、バックエンド + UI の同時実装
- **例外**: ユーザーが明示的に「まとめて」「一括で」と指示した場合のみ

### 変更範囲の上限

- **1 ターンで新規・変更するファイルは原則 3 個まで**
- **1 ターンで触る層は 1 つだけ**（例: infrastructure のターンで usecase を触らない）
- **無関係なリファクタ・ついで修正はしない**（スライス外の変更は別スライスにする）

### 着手前の確認

実装を始める前に、必ず次を行う。

1. **参照ドキュメントを読む**（存在するものから）
   - `docs/adr/` … 設計判断・採用/不採用の理由
   - `docs/tables/` … DB・RLS に触る場合
   - `docs/design/` … 画面・機能仕様
   - `docs/implementation/<feature>/` … 機能別のスライス定義（あれば）
   - 既存の `*.test.ts` … TDD で仕様が固定されている場合
2. **今回のスライスを 1 行で宣言する**（層 + 対象 + 完了条件）
3. `docs/implementation/<feature>/` が無い場合は、スライス定義を短く提示してから着手する

### 層の順序

下位層から上位層へ。上位層だけ先に実装しない。

```
domain/models → domain/repositories → infrastructure → usecase → app → presentation
```

### テスト駆動（TDD）

| 状況 | 行動 |
|---|---|
| 対象のテストが既にある | **テストを通す実装のみ**。テストの緩和・削除は禁止 |
| テストがない | 先にテスト案を提示して止まる（ユーザーが「実装のみ」と明示した場合のみ例外） |
| スライス完了時 | **そのスライス用のテストを実行**し、結果を報告する |

テスト実行例:

```bash
npm run test:run -- path/to/target.test.ts
```

### ADR との接続

| 状況 | 行動 |
|---|---|
| 既存 ADR で説明済み | その方針に従う。完了報告で関連 ADR をリンクする |
| 採用/不採用の新しい設計判断が必要 | **実装前に** ADR 草案または判断理由を提示して止まる |
| 実装詳細のみの話 | ADR は不要。コード + 短い解説で足りる |

毎スライス ADR を書く必要はない。**判断が生まれたときだけ** ADR を検討する。
検討して必要と判断した場合は私に相談すること。

## 実装順序（詳細）

1. domain/models … 型定義（最小限）
2. domain/repositories … インターフェース定義
3. infrastructure … DB アクセス実装
4. usecase … オーケストレーション
5. app … Server Action
6. presentation … UI コンポーネント

各ステップ完了後、**次へ進む前にユーザーの確認を取る**。

## UI 実装時の注意

- 既存コンポーネント（`src/presentation/components/`）のスタイルを参考にする
- カラー: emerald（プライマリ）、amber（アクセント）
- アイコン: `lucide-react` を使用
- 画像: `next/image` を使用
- shadcn/ui コンポーネント: `src/components/ui/` を活用

## スライス完了時の報告形式（必須）

説明は最小限ではなく、**レビューしやすい形で必ず報告**する。各スライス終了時に次を含める。

1. **何をしたか**（1〜3 行）
2. **なぜこの順・この層か**（層ルール・ADR との関係を 1 行）
3. **触ったファイル一覧**
4. **実行したテストと結果**
5. **レビュー時に見るポイント**（2〜3 個。セキュリティ・層の依存・仕様の観点）
6. **次のスライス提案**（1 つだけ。自動では着手しない）

最後に「次のスライスに進みますか？」と確認する。

## 出力形式

- コードは**ファイルに直接書き込む**
- スライス外の説明は完了報告に集約する（実装中はコードに集中）
- **次のスライスに自動で進まない**（ユーザー確認後のみ）

## 開始

ユーザーの要求を聞き、**スライス定義を宣言してから**実装を開始してください。
