# 実装モード

あなたは今から**実装モード**で動作します。

## このモードの目的

ユーザーの要求に対して、**実際にファイルを作成・編集**してコードを実装します。
**1 機能（操作）単位**で end-to-end に実装し、**1 PR = 1 機能**になるよう PR サイズを抑えます。

## 動作ルール

1. **直接実装**: コードをファイルに書き込む
2. **機能単位で進める**: 登録・取得・更新・削除など、**1 操作 = 1 スライス = 1 PR** を目安にする
3. **クリーンアーキテクチャ遵守**: `.cursor/rules/clean-architecture.mdc` に従う
4. **命名規則遵守**: `.cursor/rules/file-naming.mdc` に従う
5. **コードスタイル遵守**: `.cursor/rules/code-style.mdc` に従う（絵文字・過度なコメント禁止）
6. **UIデザイン遵守**: `.cursor/rules/ui-design.mdc` に従う（統一感のあるデザイン）

## 実装規律（レビュー負荷を抑えるための必須ルール）

### スライスの定義

**1 スライス = 1 機能（操作）の end-to-end 実装**。

| 操作 | スライスに含める層（必要なものだけ） |
|---|---|
| **登録** | domain（不足分）→ infrastructure → usecase → app → presentation |
| **取得** | domain（不足分）→ infrastructure → usecase → app → presentation |
| **更新** | domain（不足分）→ infrastructure → usecase → app → presentation |
| **削除・脱退** | domain（不足分）→ infrastructure → usecase → app → presentation |

- **1 スライス内では下位層 → 上位層**の順で実装する（依存方向を守る）
- **禁止**: 無関係な操作の混在（例: 登録 PR に参加・脱退を含めない）
- **例外**: ユーザーが明示的に「層だけ」「メソッドだけ」と指示した場合

### PR・ブランチの単位

- **1 機能 = 1 ブランチ = 1 PR** を原則とする
- マージ後にその操作だけ動作確認できる状態にする
- 共有の domain / repository interface は、**最初に必要になった機能 PR** で追加する

### 変更範囲

- **1 ターンで触るのは 1 機能（1 操作）のみ**
- 無関係なリファクタ・ついで修正はしない
- 機能が大きい場合は、**操作内のサブステップ**（例: app → presentation）に分けてもよいが、**同一 PR 内**で完結させる

### 着手前の確認

実装を始める前に、必ず次を行う。

1. **参照ドキュメントを読む**（存在するものから）
   - `docs/adr/` … 設計判断・採用/不採用の理由
   - `docs/tables/` … DB・RLS に触る場合
   - `docs/design/` … 画面・機能仕様
   - `docs/implementation/<feature>/` … 機能別スコープ（あれば）
   - 既存の `*.test.ts` … TDD で仕様が固定されている場合
2. **今回のスライスを 1 行で宣言する**（操作 + 完了条件）
3. `docs/implementation/<feature>/` が無い場合は、スコープを短く提示してから着手する

### 層の順序（1 機能内）

```
domain/models → domain/repositories → infrastructure → usecase → app → presentation
```

上位層だけ先に実装しない。

### テスト駆動（TDD）

| 状況 | 行動 |
|---|---|
| 対象のテストが既にある | **テストを通す実装のみ**。テストの緩和・削除は禁止 |
| テストがない | 機能実装と同時にテストを追加（usecase / infrastructure / action） |
| スライス完了時 | **関連テストを実行**し、結果を報告する |

テスト実行例:

```bash
npm run test:run -- path/to/target.test.ts
```

### ADR との接続

| 状況 | 行動 |
|---|---|
| 既存 ADR で説明済み | その方針に従う。完了報告で関連 ADR をリンクする |
| 採用/不採用の新しい設計判断が必要 | **実装前に** ADR 草案または判断理由を提示して止まる |
| 実装詳細のみの話 | ADR は不要。コード + 短い解説で足りる |

## UI 実装時の注意

- 既存コンポーネント（`src/presentation/components/`）のスタイルを参考にする
- カラー: emerald（プライマリ）、amber（アクセント）
- アイコン: `lucide-react` を使用
- 画像: `next/image` を使用
- shadcn/ui コンポーネント: `src/components/ui/` を活用

## スライス完了時の報告形式（必須）

1. **何をしたか**（1〜3 行）
2. **スコープ外にしたこと**（別 PR / 別ブランチ向け）
3. **触ったファイル一覧**
4. **実行したテストと結果**
5. **レビュー時に見るポイント**（2〜3 個）
6. **次の機能提案**（1 つだけ。自動では着手しない）

最後に「次の機能に進みますか？」と確認する。

## 出力形式

- コードは**ファイルに直接書き込む**
- **次の機能に自動で進まない**（ユーザー確認後のみ）

## 開始

ユーザーの要求を聞き、**機能スコープを宣言してから**実装を開始してください。
